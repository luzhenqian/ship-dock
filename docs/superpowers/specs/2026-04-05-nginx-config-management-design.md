# Nginx Configuration Management

## Overview

Allow users to view and edit a subset of Nginx configuration parameters per project, with changes taking effect immediately. The full generated Nginx config is shown as a read-only preview alongside the editable form.

## Scope

- Project-level only (not global)
- Editable fields: request limits (body size, timeouts) and compression/caching (gzip, proxy buffering)
- Non-editable parts (proxy_pass, SSL, server_name, etc.) remain auto-generated by the deploy pipeline
- Edit permission: OWNER, ADMIN, DEVELOPER. VIEWER can only read.

## Data Model

New `NginxConfig` table with one-to-one relation to `Project`:

```prisma
model NginxConfig {
  id        String  @id @default(uuid())
  projectId String  @unique
  project   Project @relation(fields: [projectId], references: [id], onDelete: Cascade)

  // Request limits
  clientMaxBodySize   Int     @default(10)    // MB
  proxyReadTimeout    Int     @default(60)    // seconds
  proxySendTimeout    Int     @default(60)    // seconds
  proxyConnectTimeout Int     @default(60)    // seconds

  // Compression & caching
  gzipEnabled    Boolean @default(true)
  gzipMinLength  Int     @default(1024)  // bytes
  gzipTypes      String  @default("text/plain text/css application/json application/javascript text/xml")
  proxyBuffering Boolean @default(true)
  proxyBufferSize String @default("4k")
  proxyBuffers    String @default("8 4k")

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

Project model adds: `nginxConfig NginxConfig?`

### Validation Rules

| Field | Type | Min | Max | Unit |
|-------|------|-----|-----|------|
| clientMaxBodySize | Int | 1 | 1024 | MB |
| proxyReadTimeout | Int | 1 | 3600 | seconds |
| proxySendTimeout | Int | 1 | 3600 | seconds |
| proxyConnectTimeout | Int | 1 | 3600 | seconds |
| gzipEnabled | Boolean | - | - | - |
| gzipMinLength | Int | 0 | 1048576 | bytes |
| gzipTypes | String | - | - | space-separated MIME types |
| proxyBuffering | Boolean | - | - | - |
| proxyBufferSize | String | - | - | e.g. "4k", "8k" |
| proxyBuffers | String | - | - | e.g. "8 4k" |

## Backend API

New `nginx-config` module (controller + service).

### Endpoints

#### GET `/api/projects/:id/nginx-config`

- **Auth**: JWT, VIEWER+
- **Response**: `{ config: NginxConfigDto, preview: string }`
- If no NginxConfig record exists, return default values and a preview generated from defaults
- `preview` is the full generated Nginx config file content (read-only)

#### PATCH `/api/projects/:id/nginx-config`

- **Auth**: JWT, DEVELOPER+
- **Body**: Partial `NginxConfigDto` (only include fields to change)
- **Validation**: Zod schema with the rules above
- **Response**: `{ config: NginxConfigDto, preview: string }`
- **Precondition**: Project must have a domain set. Return 400 if no domain.
- **Flow**:
  1. Validate input
  2. Upsert NginxConfig record in database
  3. Generate full Nginx config via `buildConfig()`
  4. Back up current config file
  5. Write new config to `/etc/nginx/sites-available/{slug}.conf`
  6. Run `nginx -t` to validate
  7. If valid: `nginx -s reload`, return 200
  8. If invalid: restore backup, rollback database, return 400 with nginx error

### Changes to NginxStage

- `buildConfig()` accepts NginxConfig fields and injects them into the generated config:
  - `client_max_body_size` in server block
  - `proxy_read_timeout`, `proxy_send_timeout`, `proxy_connect_timeout` in location block
  - `gzip` directives in server block (when enabled)
  - `proxy_buffering`, `proxy_buffer_size`, `proxy_buffers` in location block
- During deployment, read NginxConfig from database. If none exists, use defaults.

### Generated Config Example

```nginx
server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    client_max_body_size 10m;

    gzip on;
    gzip_min_length 1024;
    gzip_types text/plain text/css application/json application/javascript text/xml;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        proxy_read_timeout 60s;
        proxy_send_timeout 60s;
        proxy_connect_timeout 60s;
        proxy_buffering on;
        proxy_buffer_size 4k;
        proxy_buffers 8 4k;
    }
}
```

## Frontend

### Location

New tab/section in `/projects/[id]/settings` page labeled "Nginx".

### Layout

**Upper section — Edit Form:**

Two card groups:

1. **Request Limits**
   - `client_max_body_size`: number input + "MB" suffix
   - `proxy_read_timeout`: number input + "秒" suffix
   - `proxy_send_timeout`: number input + "秒" suffix
   - `proxy_connect_timeout`: number input + "秒" suffix

2. **Compression & Caching**
   - `gzip`: toggle switch
   - `gzip_min_length`: number input (shown when gzip enabled)
   - `gzip_types`: text input (shown when gzip enabled)
   - `proxy_buffering`: toggle switch
   - `proxy_buffer_size`: text input (shown when buffering enabled)
   - `proxy_buffers`: text input (shown when buffering enabled)

Bottom: "Save" button. Saving triggers PATCH and immediately applies.

**Lower section — Config Preview:**

- Read-only `<pre>` block showing the full generated Nginx config
- Monospace font with basic syntax highlighting
- Updates in real-time as the form changes (preview generated client-side)

### State Handling

- **No domain**: show message "请先部署项目并配置域名后，才能管理 Nginx 配置"
- **Save success**: toast notification
- **Save failure** (nginx -t fails): show error message from backend
- **Loading**: skeleton/spinner while fetching config

### Tech

- React Hook Form + Zod for form validation (same pattern as rest of project)
- React Query for data fetching (GET) and mutation (PATCH)
- Client-side preview generation mirrors backend `buildConfig()` logic

## Apply Flow

```
User edits form → real-time preview updates (client-side)
  → clicks Save
    → PATCH /api/projects/:id/nginx-config
      → validate params
      → upsert NginxConfig in database
      → buildConfig() generates full config
      → write to /etc/nginx/sites-available/{slug}.conf
      → nginx -t
        → pass → nginx -s reload → 200 OK
        → fail → restore old config, rollback DB → 400 + error
```

## Deploy Pipeline Integration

The existing NginxStage in the deploy pipeline reads NginxConfig from the database when generating configuration. If no record exists, default values are used. This ensures manual edits and deployments always produce consistent results from the same source of truth.
